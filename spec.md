# WBS Tool 仕様書

`index.html` 一枚で動く WBS / ガントチャートツールの仕様をまとめる。
実装の詳細な対応関係は [`CLAUDE.md`](CLAUDE.md) を参照。

最終更新: 2026-06-26

## 1. 概要

- HTML/CSS/JS をすべて内包した**単一ファイル** `index.html`。ブラウザで開くだけで動作する。
- サーバー・ビルド・パッケージ管理・依存ライブラリは**一切なし**。
- データはブラウザの `localStorage` に保存。**外部送信は行わない**。

## 2. データモデル

状態は `tasks[]`（タスクの単一配列）と採番カウンタ `uid` で表現する。

### タスク（`tasks[]` の要素）

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `id` | number | 一意ID。`uid++` で採番（再利用しない） |
| `level` | number | 階層の深さ（0=大項目）。親子関係は level で表現 |
| `name` | string | タスク名 |
| `assignee` | string | 担当者 |
| `start` | string | 開始日（`YYYY-MM-DD`）。空可 |
| `end` | string | 終了日（`YYYY-MM-DD`）。空可 |
| `progress` | number | 進捗率 0–100 |
| `status` | string | `notstarted` / `inprogress` / `done` / `hold` |
| `note` | string | 備考メモ |
| `deps` | number[] | **先行タスクの id 配列**（Finish-to-Start・葉タスク間限定） |
| `collapsed` | boolean | 子タスクを折りたたみ表示中か |

### 階層の表現

- 親子関係は `level` の大小で表す。ある行の子は「直後から `level` がより大きい行が続く範囲」。
- 親タスクの開始/終了/進捗は葉ノードから**自動集計（ロールアップ）**し、手入力値は無視される。

### 永続化（localStorage キー）

| キー | 内容 |
|------|------|
| `wbs-tool-data-v1` | `{ uid, tasks }` |
| `wbs-tool-colwidths-v1` | テーブル列幅の配列 |

旧データの後方互換: 読込時に `deps` 欠落を `[]` で補完する（`ensureDeps()`）。

## 3. 機能

### 3.1 WBSテーブル（左ペイン）

- 階層タスクの追加／削除／インデント／アウトデント／並べ替え（`addRoot` / `addChild` / `addSibling` / `indent` / `outdent` / `moveUp` / `moveDown` / `deleteTask`）。
- WBS番号（`1.2.3`）を自動採番（`wbsNumber`）。
- 親行の開始／終了／進捗は子から集計（`summaryOf`）し、淡色のread-only表示。
- 削除は「元に戻す」トースト（直前1回分）に対応。
- 列幅はヘッダー右端のドラッグで変更可（操作列を除く）。ダブルクリックで既定幅に戻す。

### 3.2 ガントチャート（右ペイン）

- `DAY_W` px/日でバーを描画。土日シェーディング・今日の線・マイルストーン（開始＝終了）・親のサマリーバー。
- 初回ロード時のみ「今日」をビュー中央へスクロール。
- テーブルとガントは**縦スクロールのみ同期**（横は独立）。

### 3.3 タスク依存関係（先行/後続）※ #22 で実装

- 各葉タスクに先行タスク（`deps`）を設定できる。種別は **Finish-to-Start (FS) のみ**。
- テーブルの「先行」列で、先行タスクを **WBS番号チップ**で表示。クリックで削除、`＋` セレクトで追加。
- ガント上に**依存矢印**（先行バー終端 → 後続バー始端、L字＋矢じり）を SVG オーバーレイで描画。折りたたみで非表示の端点はスキップ。
- **循環参照対策**: 候補から循環する相手を除外し、追加時にもガード（`depsCreatesCycle`）。
- 親タスクは依存を持たない（サマリーのため）。

### 3.4 集計・入出力

- ヘッダーに全体進捗・タスク数・ステータス別件数（葉タスク基準）を表示（`renderStats`）。
- JSON出力／読込（`exportJSON` / `importJSON`）、CSV出力（Excel対応・BOM付き、「先行」列を含む）。
- サンプルデータ読込（`loadSample`）。

## 4. 主要関数

| 関数 | 役割 |
|------|------|
| `render()` | `renderTable()` + `renderGantt()` + `renderStats()` + `save()` |
| `childrenRange(idx)` | 子タスクの範囲 `[start, endExclusive)` を返す |
| `summaryOf(idx)` | 親タスクの開始/終了/進捗を葉から集計 |
| `wbsNumber(idx)` | 階層番号を採番 |
| `depsCreatesCycle(currentId, predId)` | 依存追加が循環になるか（DFS） |
| `ensureDeps()` | 全タスクに `deps` を保証（後方互換） |

## 5. ロードマップ

### クリティカルパス可視化（#14・優先度: 低／backlog）

依存モデル（#22）が前提だったが解消済み。日付は手入力で固定する設計のため、**フルCPM（ES/LS/float）ではなく軽量版にリスコープ**:

- 最長依存チェーン（クリティカルパス相当）の強調表示
- 各タスク→後続までの **隙間（slack 日数）** 表示
- **日付逆転の警告**（後続が先行より前に開始）
- 強調表示の ON/OFF トグル

着手トリガー: 分岐・合流を含む計画を日常的に扱う／タスクが大規模化し目視追跡が困難になったとき。

## 6. 設計方針

- **単一ファイル構成を崩さない**（外部依存・ビルド工程を持ち込まない）。
- 機能単位で Issue → ブランチ → PR → `/code-review` のワークフローを守る（詳細は [`CLAUDE.md`](CLAUDE.md)）。
